系统验收

发表于2026-03-23|更新于2026-04-09

系统验收汇报讲稿：前端工程化与性能优化

用途：答辩/验收时口头汇报，可按时间删减。
工程：discountmall-vue3-CBD / discountmall-vue3-CBD-UP（移动端）、discountmall-web-CBD / discountmall-web-CBD-UP（桌面端）；后端 Spring Boot 与前端通过 /wx 代理联调。
详细代码对比：见 Vue3-CBD-UP-优化代码前后对比.md、毕设-系统验收演讲稿-优化对比版.md。

一、开场（约 1 分钟）

各位老师好。我本次毕设的前端部分，围绕小商城用户端展开：在 Vue 3 + Vite 技术栈上完成业务功能，并在此基础上做了工程化组织与性能及稳定性优化。

移动端采用 Vant，桌面 Web 端采用 Element Plus，接口层统一使用 axios；优化前、优化后分别以 *-CBD 与 *-CBD-UP 两套工程对照，便于说明改动了什么、解决了什么问题。

下面我从工程化开发和性能优化两部分汇报。

二、前端工程化开发（约 3～4 分钟）

2.1 为什么采用 Vue 3 + Vite

  • Vue 3：组合式 API 便于把「请求、状态、生命周期」集中在一处书写，页面复杂时比选项式更易维护。
  • Vite：开发阶段冷启动快、按需编译；生产构建基于 Rollup，适合现代 ES 模块，和 Vue 3 生态匹配度高。

2.2 工程目录与职责划分

典型结构包括：

  • **src/views**：按业务拆页面（首页、分类、搜索、详情、用户等），路由与文件夹对应，便于定位。
  • **src/router**：路由表集中配置；配合 () => import() 做路由级懒加载（见下一节性能部分）。
  • **src/api**：把接口路径、方法封装成函数，页面只调 getHome、goodsDetail 等，避免 URL 散落在各组件。
  • src/utils：如 request.js 封装 axios 实例、拦截器、统一错误与 Token，属于横切关注点，业务代码不重复写鉴权与 baseURL。
  • **src/stores**（Pinia）：跨页面共享状态（如应用级配置），与组件内 ref 本地状态区分使用场景。
  • vite.config.js：@ 路径别名、scss 全局变量注入、开发代理 /wx → 后端，实现前后端端口分离下的本地联调。

这样划分后，改接口只动 api/utils，改页面只动 views，改构建只动 Vite 配置，符合工程化「高内聚、低耦合」的目标。

2.3 标准化脚本与交付形态

各前端工程 package.json 中统一提供：

  • **npm run dev**：本地开发，热更新。
  • **npm run build**：生产构建，产出 dist，用于静态部署或验收时展示「上线形态」。
  • **npm run preview**：预览构建结果（Vue3-UP 中 preview 与 dev 共用代理配置，避免预览与开发行为不一致）。

依赖通过 package.json / lock 文件锁定版本，保证多人或换机器可复现构建。

2.4 联调与部署层面的工程化

  • 浏览器只访问前端端口（如 6256），接口统一走相对路径 **/wx/...**，由 Vite 代理到后端（如 9090），避免写死后端地址、也方便切换环境。
  • Token 等在 request 拦截器统一注入，减少漏带凭证导致的偶发错误。

小结（可背诵一句）：工程化体现在 「目录职责清晰 + 构建脚本统一 + 请求与路由集中封装 + 代理解耦后端」。

三、性能与稳定性优化（约 4～5 分钟）

优化目标可以概括为：减轻首屏与重复请求压力、降低高并发与弱网下的错误与乱序、改善用户可感知的加载体验。下面按层次说明「做了什么、为什么有效」。

3.1 加载路径：路由懒加载与页面缓存

  • 路由懒加载：使用 component: () => import('@/views/...')，按路由拆 chunk，首屏不必下载全部页面脚本，有利于首包体积与首次可交互时间。
  • **<keep-alive> + meta.keepAlive**（移动端 App.vue）：Tab 类页面切换时减少重复挂载，配合后续「离页取消请求」，避免无意义的重复请求与闪烁。

3.2 请求层（axios）：稳链路、少误判、可控重试

文件：**src/utils/request.js**（Vue3-UP / Web-UP 策略略有不同，答辩时可说明「两端场景不同」）。

  • 超时调整：适当放宽客户端 timeout，减少「后端仍在处理、客户端先断开」的误判。
  • 业务码 501（未登录）节流：短时间窗口内多次 501 只处理一次跳转/提示，避免并发请求触发重复 Toast 与重复跳转。
  • 网络错误分级提示：区分超时、无响应（多为断网）、5xx、404 等，替代「一律连接超时」，利于用户理解与问题定位。
  • GET 有限重试：
    • Vue3-UP：仅 HTTP 502/503 时最多补一次，并带短随机延迟，避免压测或故障时重试风暴。
    • Web-UP：侧重超时/无响应类 GET 的有限重试；写操作（加购、下单）不自动重试。

3.3 API 层：短 TTL 缓存与并发去重（削峰）

文件：**src/api/api.js**（主要在 Vue3-UP）。

  • 首页 getHome：内存 TTL（如约 20 秒）+ in-flight Promise 复用：短时间重复进入首页或并发触发时，不重复打聚合接口，削峰。
  • 商品详情、品牌列表/详情等：Map 按 id 或参数 key 缓存 + pending 去重，同一 key 并发只发一次 HTTP。
  • 支持 **skipCache、signal**：手动重试/强制刷新与取消请求时行为正确，避免脏缓存。

3.4 页面层：取消竞态、防连点、可感知性能

  • AbortController：首页、详情、搜索等场景，在 组件卸载、路由参数变化、搜索词变化时 abort，避免旧请求晚到覆盖新数据（典型竞态问题）。
  • 骨架屏（Vant Skeleton）：main.js 注册后，首屏先占位再填数据，提升可感知速度，减少用户因白屏反复刷新。
  • 失败态与重试入口：如首页统一 loadHome，失败可点重试并 skipCache 拉最新。
  • 防连点（busy / loading 锁）：加购、领券、收藏等，请求进行中禁止重复提交，降低重复写请求与后端冲突。
  • 详情页 watch 路由参数：同一组件实例切换商品 id 时清空旧数据并重拉，避免显示上一商品。

3.5 构建与代理层（Vue3-UP）

文件：**vite.config.js**。

  • 对 /wx 代理增加 较长超时、使用 Node http.Agent Keep-Alive 与 maxSockets 等，减轻开发/压测时 Vite 代理进程与后端之间频繁建连、代理过早断开的问题。
  • 说明：使用 JMeter 直连前端端口时，请求走 Node 代理链路，与浏览器内页面逻辑不同；汇报时宜区分代理层优化与浏览器内 axios/缓存优化各自作用范围（详见 vue3-CBD与CBD-UP优化对比与实现说明.md）。

3.6 桌面 Web 端补充（Web-UP）

  • 同域请求并发上限队列：避免单页瞬间占满浏览器连接，弱网下关键请求仍有机会完成。
  • **路由切换 refreshNavigationAbort**：与 axios signal 结合，换页时取消未完成请求，与移动端 Abort 思路一致。

小结（可背诵一句）：性能优化是 「加载路径（懒加载/缓存页）→ 请求与数据（削峰/取消/重试）→ 交互（骨架/防连点）→ 代理与链路（Vue3-UP）」 分层推进。

四、优化前后对照与验收表述（约 1 分钟）

  • 对照方式：同一功能在 *-CBD 与 *-CBD-UP 中对比 request.js、api.js、关键页面与 vite.config.js 即可说明差异。
  • 数据对比：仓库 report/ 与 WEB与Vue3-report-Old-New对比汇总.md 中整理了 JMeter 统计；汇报时可选取 错误率、P95/P99、吞吐 等 1～2 个指标口述，并说明测试条件（线程数、样本量、是否同一脚本）。
  • 注意：不要把「浏览器内优化」与「仅走代理的压测路径」混为一谈，老师追问时按文档中的归因回答即可。

五、演示说明（约 30 秒，可选）

演示顺序：先启后端 jar，再 npm run dev 启一个前端，浏览器访问 http://localhost:6256；四个前端端口相同，一次只开一个。具体命令见 **毕设-工程启动路径与命令说明.md**。

六、结束语（约 30 秒）

综上，本课题在前端侧完成了模块化工程搭建与可复现的构建流程，并针对重复请求、并发与弱网、竞态与体验做了分层优化，形成 CBD / CBD-UP 对照，便于验收与论文撰写。以上是我的汇报，谢谢各位老师。请老师批评指正。

七、创新点（可单独汇报 1 分钟）

7.1 本课题创新点（系统开发类口径）

  1. 构建了可复现实验对照体系（CBD vs CBD-UP） \n
    不是只展示“优化后结果”，而是保留优化前后两套可运行工程，并在同脚本下进行压测对比，使优化结论具备可追溯性与可复现性。

  2. 提出并落地请求层“稳态化策略中心” \n
    在 request.js 统一承载 Token 注入、501 节流、错误分级、有限重试与取消信号，减少页面散乱实现，提升一致性与可维护性。

  3. 实现读多接口“短 TTL + 并发去重”的前端削峰机制 \n
    在 api.js 对首页聚合与详情类接口引入缓存有效期与 in-flight Promise 复用，在不改后端协议前提下降低重复请求与瞬时压力。

  4. 形成页面级竞态治理模式（Abort + 状态流转） \n
    将 AbortController 系统化应用于首页、搜索、详情等关键路径，避免旧请求晚到覆盖新状态，减少异步乱序引发的数据错误。

  5. 将交互体验优化与稳定性优化协同设计 \n
    通过 Skeleton、失败重试入口、busy/防连点等机制，既提升可感知加载体验，也降低用户高频误操作导致的重复写请求。

  6. 在代理链路侧进行工程级优化（Vue3-UP） \n
    对 /wx 代理增加超时与连接复用（Keep-Alive/套接字参数），并在报告中明确其作用路径，体现从页面到链路的分层优化思路。

7.2 一句话创新总结（答辩可直接说）

本课题的创新不在于提出新算法，而在于构建了可复现对照 + 分层优化闭环：从工程结构、请求策略、接口削峰、页面竞态到代理链路，形成了可运行、可对比、可验证的系统开发方案。

附录：可能的追问与短答（备查）

问题 短答方向
工程化体现在哪？ 目录职责、api/utils/router 分层、Vite 脚本与代理、依赖锁定。
为什么用懒加载？ 减首包、按路由拆 chunk。
为什么缓存首页又要 TTL？ 削峰且控制数据新鲜度；重试可用 skipCache。
为什么 GET 才自动重试？ 相对幂等；POST 重复可能重复下单/加购。
AbortController 解决什么？ 旧请求晚到导致数据错乱、离页仍更新 UI。
Web 与 Vue3 重试为何不同？ 场景与风险不同：一个偏网关 502/503，一个偏超时与连接。
