discountmall-web-CBD 模块化与组件化实现说明

本文不再做项目对比，重点回答一个问题：discountmall-web-CBD 是如何在代码层面实现模块化与组件化开发的。
并且会明确说明每个核心文件夹、关键文件分别负责什么。

一、项目整体如何体现“模块化”

在 discountmall-web-CBD 中，模块化不是一句概念，而是通过“按职责分目录 + 按能力分文件”落实的。
核心思路是：页面模块、布局模块、数据模块、状态模块、工具模块分开维护。

项目文件目录（当前）

1）src/views：业务页面模块

这是业务模块主战场，每个子目录代表一类业务场景：

  • src/views/home：首页相关页面与展示逻辑。
  • src/views/items：商品相关页面（分类、搜索、详情、专题等）。
  • src/views/order：购物车、结算、订单详情等交易流程页面。
  • src/views/user：个人中心、地址、优惠券、订单列表等用户域页面。
  • src/views/login：登录、注册、忘记密码等身份认证页面。

这类目录结构体现了“按业务域拆模块”，便于后续按功能团队并行开发。

2）src/router：路由模块

  • src/router/index.js 统一定义页面路由、参数、鉴权元信息（如 meta.login）、页面缓存策略（meta.keepAlive）。
  • 它把 URL 与页面模块连接起来，承担“页面编排中枢”角色。

也就是说，路由模块负责“去哪一个页面”，而页面模块负责“页面内部怎么实现”。

3）src/api：接口访问模块

  • src/api 目录中按业务封装请求方法（如商品、购物车、用户、订单接口）。
  • 页面不会直接散落写 axios 调用，而是通过 API 模块拿数据。

这样做的好处是：后端接口变更时，只需优先调整 API 层，减少对页面层的冲击。

4）src/stores：状态管理模块

  • 基于 Pinia 管理全局状态（例如购物车数量、应用级状态）。
  • 页面与布局组件可以通过 store 共享状态，而不是层层传递。

这避免了复杂页面中“props 传递链过长”的问题。

5）src/utils：通用工具模块

  • 放与业务弱耦合的通用能力，如本地存储封装（local-storage）、格式转换等。
  • 页面和组件调用统一工具函数，减少重复代码。

6）src/assets：资源与样式模块

  • src/assets/scss 存放全局变量、mixin、基础样式。
  • 通过构建配置注入变量，使各组件样式保持一致设计语言。

这让样式也具备模块化能力：设计规范集中维护，业务样式就近定义。

二、项目整体如何体现“组件化”

组件化的关键不是“把文件拆小”，而是让每个组件有清晰职责、可复用边界和稳定接口。

1）布局组件化：src/layouts

当前核心布局在以下文件中完成职责拆分：

  • src/layouts/WebLayout.vue
    作为布局容器，负责“组装”页面结构，不堆叠太多业务细节。

  • src/layouts/components/WebHeader.vue
    负责顶部区域（Logo、搜索、登录态、用户下拉、购物车入口）。

  • src/layouts/components/WebSidebar.vue
    负责左侧导航菜单与当前路由高亮。

  • src/layouts/components/WebContent.vue
    负责主内容渲染（router-view）和 keep-alive 缓存策略承接。

这就是典型的“父组件编排 + 子组件职责单一”组件化模型。

2）通用组件：src/components

  • 放可跨页面复用的组件（如通用头部、导航、标签条等，视当前项目实际组件而定）。
  • 这些组件不应与某个单一业务页面强绑定，强调复用性。

3）页面内组件拆分

在较复杂页面中，可继续采用“页面容器 + 子模块组件”方式拆分，例如：

  • 商品详情页拆成：轮播区、规格选择区、推荐商品区。
  • 用户中心页拆成：用户信息卡片、订单入口模块、功能菜单模块。

这种拆分能让单文件复杂度下降，逻辑更清晰，后续改版更稳。

三、关键入口文件做了什么（非常重要）

1）src/main.js

应用启动入口，主要完成：

  • 创建 Vue 应用实例；
  • 注册 Pinia 状态管理；
  • 挂载路由；
  • 注册 UI 组件库（Element Plus）；
  • 引入全局样式。

它相当于应用的“总装配车间”。

2）src/App.vue

根组件，当前职责很薄（通常只承载 router-view 或最外层结构）。
这是一种良好实践：让 App 保持轻量，把复杂结构下沉到布局层与页面层。

3）vite.config.js

构建层配置文件，主要作用：

  • 配置路径别名（如 @ -> src）；
  • 配置 SCSS 预处理注入（全局变量、mixin）；
  • 配置开发服务端口与代理。

它保证了模块开发体验（短路径引用、统一样式能力）。

四、一个“从请求到页面”的模块协作流程示例

以“进入商品列表页”为例：

  1. 用户访问路由 /items；
  2. src/router/index.js 将路由匹配到 src/views/items/CatalogPage.vue；
  3. 页面调用 src/api 中商品相关接口函数请求数据；
  4. 若涉及全局状态（如购物车计数），通过 src/stores 同步；
  5. 页面渲染在 WebLayout -> WebContent 的内容区域中；
  6. 样式由页面局部样式 + src/assets/scss 全局能力共同生效。

这个流程说明：每一层都做自己该做的事，互相配合但不过度耦合。

五、为什么这种实现方式有效

因为它同时解决了前端项目常见三大问题：

  • 文件膨胀：通过目录分层和组件拆分控制单文件复杂度；
  • 重复开发：通过通用组件与工具模块实现复用；
  • 改动风险高：通过模块边界隔离影响范围，减少“牵一发动全身”。

六、总结

discountmall-web-CBD 的模块化体现在“按业务与能力分层组织代码”；
组件化体现在“将布局与页面拆成职责清晰、接口明确、可复用的 Vue 组件”。
两者结合后，项目更适合长期维护、多人协作和持续迭代。
